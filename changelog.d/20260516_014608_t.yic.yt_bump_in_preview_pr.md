### Changed

- Bring back the "bump version files in the Changelog Preview PR" behavior — `collect_for_release` now calls `provider.apply(level, tag=False, commit=False, push=False)` after `scriv collect`, so the preview-PR commit captures both the new `CHANGELOG.md` entry and the version-file bump for file-based providers (`uv`, `bump-my-version` with `[tool.bumpversion]`, `hatch` with a non-`vcs` source, `shell`). For tag-only providers (`bump-my-version` with no `[tool.bumpversion]`, `hatch-vcs`) `apply()` is a no-op and there's still nothing in the preview PR besides the changelog edit. This was the v0.5.0–v0.5.1 behavior, with the v0.5.x bug (tag-only providers crashing at tag time) fixed by the next item.

### Fixed

- `tag_release` now reads the version to tag from `latest_changelog_version()` instead of `provider.next(level)` / `provider.current()`. The changelog entry's title is what `scriv collect` just wrote in the preview PR — it IS the version we're releasing. This is provider-agnostic: for file-based providers the preview PR also bumped the version file to the same value, and for tag-only providers nothing else needed bumping, so the two paths converge on the right tag without any provider-style detection at tag time. This fixes the v0.5.x bug where tag-only providers crashed with `fatal: tag 'vX.Y.Z' already exists` (because `provider.current()` still reported the previous tag at merge time) without re-introducing the v0.6.x drift-between-file-and-tag for file-based projects.
